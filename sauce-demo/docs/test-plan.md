## 1. 项目背景与测试目标

### 1.1 背景

Sauce Demo（Swag Labs）是公开电商演示站点，用于练习完整测试流程。本次 Day 1 聚焦手工端到端功能验证，覆盖登录至订单完成的核心购物链路。

### 1.2 测试目标

1. 验证主流程可走通：登录 → 浏览商品 → 加购 → Checkout → 完成 → 登出。
2. 覆盖登录、商品列表、购物车、结账/订单、会话等关键模块的正向与异常场景。
3. 发现并记录真实缺陷或明确体验问题，形成可追溯的缺陷清单。
4. 输出可归档的测试计划、用例执行结果与执行统计草稿。

---

## 2. 测试范围

### 2.1 测什么（In Scope）

| 模块 | 覆盖要点 |
|------|----------|
| 登录 | 正确/错误凭据、空输入、锁定用户、非法用户名 |
| 商品列表 | 列表展示、排序（名称/价格）、进入商品详情 |
| 购物车 | 加购、数量角标、移除、进入结账 |
| 结账与订单 | 信息填写、必填校验、取消返回、完成页、返回商品页/登出 |
| 会话/权限 | 未登录访问内部页拦截；刷新后会话行为（抽样） |

### 2.2 不测什么（Out of Scope）

- API / 后端接口（留待后续阶段）
- UI 自动化脚本编写与回归套件（留待后续阶段）
- 移动端原生 App、多浏览器深度兼容矩阵
- 支付网关真实扣款、性能压测、安全性渗透
- 账号注册、第三方登录（站点无此能力）

---

## 4. 测试策略

| 策略 | 说明 |
|------|------|
| 功能测试 | 按模块设计用例，优先保证 P0 主路径 |
| 异常测试 | 错误密码、空字段、锁定用户、未登录直达等 |
| 兼容性 | **仅 Chrome**；不扩展到 Firefox/Safari/Edge |
| 执行顺序 | 先全部 P0，再 P1/P2；失败项截图归档至 `evidence/` |
| 数据策略 | 使用站点公开演示账号；每次执行尽量从干净登录态开始 |

---

## 5. 准入 / 准出标准

### 5.1 准入

- 演示站可访问，登录页正常加载。
- 至少具备 `standard_user` 可用凭证。
- 测试计划、用例文档已就绪可执行。

### 5.2 准出（Day 1）

- P0 用例全部执行完毕。
- 无未关闭的**致命**缺陷阻塞主流程（演示站不可由测试方修复时，须在报告中标明「已知问题/建议项」及影响）。
- 用例总数 ≥ 20，且均已填写实际结果与状态。
- 缺陷记录 ≥ 3 条，字段完整。
- 执行统计写入 `docs/test-report.md` 草稿。

---

## 6. 风险说明

| 风险 | 影响 | 应对 |
|------|------|------|
| 演示站随时变更 UI/文案/账号 | 用例步骤或预期失效 | 以页面实际表现为准，失败时更新用例并留档 |
| 公网依赖 / 站点不稳定 | 超时、偶发失败 | 重复执行 1～2 次；仍失败则记阻塞并注明环境 |
| `problem_user` / `error_user` 等刻意异常账号 | 易与「产品缺陷」混淆 | 在缺陷中写明账号与复现条件，标注为演示特性或体验问题 |
| 无后台数据重置能力 | 购物车等状态可能受上次操作影响 | 用登出/重新登录或新会话隔离 |

## 8. 功能模块清单（熟悉系统结论）

| 序号 | 模块 | 说明 |
|------|------|------|
| 1 | 登录 | 用户名/密码登录，错误提示，锁定用户拦截 |
| 2 | 商品列表 | Inventory 展示、排序下拉、加购入口 |
| 3 | 商品详情 | 从列表进入单个商品页（名称、价格、加购） |
| 4 | 购物车 | 角标数量、条目管理、移除、Checkout 入口 |
| 5 | 结账 | Your Information → Overview → Finish |
| 6 | 订单完成 | 成功页文案与返回入口 |
| 7 | 登出 | 侧边菜单 Logout，回到登录页 |
